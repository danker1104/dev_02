# PRD - 냉장고 알리미 (MVP)

## 1. 문서 목적
본 문서는 [기획서.md](기획서.md)와 [아키텍처.md](아키텍처.md)를 기반으로 MVP 제품 요구사항과 구현 범위를 확정한다.

## 2. 제품 개요
- 제품명: 냉장고 알리미
- 플랫폼: 웹 기반 애플리케이션(PWA)
- MVP 사용자 범위: 개인/단일 기기
- 핵심 가치: 유통기한 관리와 재고 검색으로 폐기/중복 구매 감소

## 3. 문제 정의
- 식재료를 구매 후 잊어버려 유통기한 경과 폐기가 빈번하게 발생한다.
- 장보기 중 현재 보유 재고를 정확히 몰라 중복 구매 또는 미구매가 발생한다.

## 4. 목표
- D-3 알림으로 폐기율을 낮춘다.
- 재고 검색으로 중복 구매를 줄인다.
- 자동 인식 + 수동 보정으로 등록 완료율을 높인다.

## 5. 비목표 (MVP 제외)
- 가족 실시간 동기화
- 로그인/계정 체계
- 고급 개인화 추천
- 마이크로서비스 분리

## 6. 사용자 및 사용 맥락
- 대상: 식재료 관리가 번거로운 사용자(초기 1인 가구 중심)
- 사용 맥락: 장보기 직후 등록, 외출 중 재고 확인, 유통기한 임박 확인 후 소비/폐기 처리

## 7. 핵심 사용자 흐름
1. 사용자는 카메라로 식재료를 등록한다.
2. 시스템은 식재료명/유통기한을 인식하고, 사용자는 저장 전 수정한다.
3. 시스템은 유통기한 3일 전에 푸시를 1회 발송한다.
4. 사용자는 검색으로 보유 여부를 확인한다.
5. 사용자는 수량 차감 또는 소진/폐기 처리한다.

## 8. 기능 요구사항
### FR-1 식재료 등록
- 카메라 기반 자동 인식 제공
- 자동 인식 실패 시 수동입력 전환
- 저장 전 확인/수정 모달 제공

### FR-2 재고 저장/조회
- 텍스트 데이터만 저장 (이미지 원본 미저장)
- 필수 저장 필드: 식재료명, 유통기한, 보관 구분, 단위, 용량, 수량
- 대시보드에서 신선/임박/위험 상태 시각화

### FR-3 중복 및 동일성 처리
- 동일성 기준: 식재료명 + 보관 구분 + 단위 + 용량
- 동일 항목 재등록 시 수량 증가
- 단위/용량 누락 시 신규 저장 후 병합 여부 선택

### FR-4 알림
- 유통기한 3일 전 푸시 1회
- 등록 시점 잔여 3일 이하 항목 즉시 주의 표시
- 푸시 권한 거부 시 앱 내 배너/모달로 대체 안내
- 중복 발송 방지 로그 저장

### FR-5 검색 및 재고 상태 전환
- 식재료명 검색 시 보유 여부/수량/임박 정보 반환
- 상태: 보유, 임박, 위험, 소진, 폐기
- 수량 차감, 전량 소진, 폐기 처리 지원

## 9. 정책 요구사항
- 비로그인 운영: device_id 기준 데이터 격리
- 데이터 정책: 이미지 원본 저장 금지, OCR 결과 텍스트만 저장
- 데이터 이전: 재설치/기기 변경 시 초기화 가능성 온보딩 고지

## 10. 아키텍처 요구사항 (MVP 최소)
### AR-1 시스템 구조
- 모듈러 모놀리식
- Frontend: Next.js PWA
- Backend: FastAPI
- Storage: Azure Storage Account (Table + Queue)

### AR-2 백엔드 모듈
- Device, Inventory, OCR, Notification, Search, Scheduler

### AR-3 배포 최소 단위
- web 컨테이너 1개
- api 컨테이너 1개
- worker 컨테이너 1개
- Azure Storage Account 1개

## 11. 데이터 요구사항
- DeviceTable: 기기 식별 및 푸시 권한 상태
- InventoryItemTable: 품목 마스터 및 합산 수량
- InventoryBatchTable: 유통기한 단위 배치
- InventoryEventTable: 등록/차감/소진/폐기 이벤트
- NotificationLogTable: D-3 발송 로그

## 12. 비기능 요구사항
- 성능: 대시보드/검색 API는 일반 네트워크 환경에서 체감 지연이 낮아야 한다.
- 신뢰성: D-3 알림은 동일 배치에 대해 중복 발송되지 않아야 한다.
- 보안/개인정보: 최소 수집 원칙 준수, 이미지 원본 저장 금지.
- 운영성: 실패 로그와 KPI 이벤트 수집이 가능해야 한다.

## 13. KPI
- 등록 성공률
- 등록 완료 시간
- 알림 후 24시간 재방문율
- 검색 사용률
- 소진 전환율
- 폐기율

## 14. 수용 기준
- 등록 -> 저장 -> 검색 -> 소진/폐기 -> D-3 알림이 end-to-end 동작한다.
- 자동 인식 실패 케이스도 수동입력으로 등록을 완료할 수 있다.
- 동일성 규칙이 일관되게 적용된다.
- 푸시 권한 미허용 사용자는 대체 안내를 확인할 수 있다.
- KPI 이벤트를 분석 가능한 형태로 수집한다.

## 15. 리스크 및 대응
- OCR 오인식: 저장 전 수정 UX 및 저신뢰도 경고
- 등록 피로: 빠른 수동입력 동선 제공
- 알림 미도달: 발송 로그 추적 + 앱 내 대체 안내
- Table 조회 제약: PartitionKey/RowKey 중심 조회 패턴 우선 설계
